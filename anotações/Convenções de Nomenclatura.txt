                                   CONVENÇÕES DE NOMENCLATURA

 Há diversas maneiras de aplicar boas práticas no código, uma delas é a conversão de nomenclatura, que são recomendações para nomear identificadores ajudando a tornar seu código mais limpo e fácil de compreender, mantendo um padrão para o leitor e até mesmo podendo fornecer informação sobre a função do identificador.

 As convenções de nomenclatura que ditam como identificares de variáveis, funções, classe e outras entidade são 
nomeados e formatados. Diferentes estilos de caso são usados para melhorar a legibilidade e aderir ás diretrizes
de várias linguagens de programação, frameworks ou projetos.


* Aqui está uma visão de alguns estilos de caso de programação comuns:


- Flat case: 
 Todas as letras do identificador são minúsculas, sem espaços entre as palavras.
Não é comumente usada devido á  baixa legibilidade na leitura, especialmente em nomes mãos longos.

 Uso comum: Usado as vezes para nome de domínio ou identificadores de sistema onde espaços e caracteres especiais são restritos.

 Exemplo: convencoesdenomenclatura


- Title case:
 Cada palavra do identificador começa com uma letra maiúscula, e o espaço e usado para separar as palavras.

 Uso comum: embora não seja comumente usado no código real devido ao caractere espaço, ele é frequentemente usado
em nome de arquivos, projetos ou títulos na documentação.

 Exemplo Convecoes De Nomenclatura


- Camel case:
 As palavras do identificador são unidas sem espaços, cada palavra começa com uma letra maiúscula, exceto a primeira palavra do identificador.

 Uso comum: Comumente usado para nome de variáveis e funções.

 Exemplo: convencoesDeNomenclatura


- Upper Camel case ou Pascal Case:
 Variante da "Camel case", com todas as primeiras letras das palavras do identificador são maiúsculas, incluindo a 
primeira palavra do identificador.

 Uso comum: usado pra nome de Classe comumente.

 Exemplo: ConvencoesDeNomenclatura


-  Snake case:
  As palavras do identificador são separadas por um underline (_) e, normalmente, todas as letras são minúsculas.

 Uso comum: Amplamente utilizado em linguagens como Python e Ruby pra nomes de variáveis, nome de funções e nomes
de arquivos.

 Exemplo: convencoes_de_nomenclatura 


- Screaming Snake case:
 Uma variante de "Snake case" onde todas as letras do identificador são maiúsculas. As palavras são separadas por 
underline(_).

 Uso comum: frequentemente usado pra constantes e, linguagens de programação pra distingui-las das variáveis.

 Exemplo: CONVENCOES_DE_NOMENCLATURA


- Kebab case:
 Envolve separar as palavras do identificador com hifens(-), e todas as letras normalmente são minúsculas.

 Uso comum: Não é comumente usado em linguagens de programação como identificador, porque hifens podem ser interpretados como o operador de subtração. No entanto, é predominante em URLs, nome de classe CSS e nomes de arquivos.
 
 Exemplo: convencoes-de-nomenclatura


- Screaming  Kebab case:
 Variante de "Kebab case", que as palavras do identificador e separado com hifens(-), e todas as letras em maiúsculas. 

 Uso comum: frequentemente usado pra constantes em, linguagens de programação que permitem, pra distingui-las das variáveis.



* Notas sobre a aplicação de um tipo de case:

- A escolha o estilo de caso pode depender da linguagem de programação, dos padrões de codificação da equipe com qual está trabalhando ou preferência pessoal.

- Consistência é a chave. Uma vez que um estilo de caso é escolhido para um tipo particular de identificador (por
exemplo, variável, classe), é uma prática recomendada mantê-lo durante todo o processo do código.

- Alguns ambientes de programação são mais sensíveis a maiúsculas e minúsculas, enquanto outros não. Este é outro fator que influencia a escolha e a consistência do uso do estilo de maiúscula e minúsculas.

- Ao trabalhar em equipe ou contribuir para projetos de código aberto, aderir ás diretrizes de estilo de caso estabelecida é importante para manter a legibilidade e a consistência do código.

 
 As convenções de nomenclatura preferidas podem varias com base na linguagem de programação, prática da comunidade e, as vezes, até mesmo nas estruturas ou bibliotecas específicas sendo usadas dentro da linguagem.

Exemplo: O guia de estilo do Python (PEP 8) indica o "Snake Case" pra variáveis e funções, e o "Screaming Snake Case" para as constantes.

 Essas são as convenções gerais, mas é sempre melhor consultar o guia do estilo específico ou os padrões de comunidade para a linguagem com a qual está trabalhando, pois as práticas podem evoluir com o tempo.
